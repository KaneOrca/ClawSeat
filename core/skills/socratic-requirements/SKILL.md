# Socratic Requirements

通过结构化澄清，把模糊意图翻译成可执行的需求摘要。

这个 skill 只做四件事：
- 读 `capability-catalog.yaml` 匹配最相关的下游能力
- 对声明了 `template_index_path` 的 skill 先做模板匹配
- 按匹配条目的 `required_fields` 动态生成问题
- 输出 skill 专用摘要合同或通用摘要

它不负责：
- 收集执行素材
- 设计具体工作流步骤
- 运行下游 skill
- 读取代码或做实现

## 何时使用 / 何时跳过

**使用**:
- 请求模糊、关键信息缺失或存在多种解读
- 任务横跨多个层但边界不清
- 需要向下游 skill / ACP 委托链路交付稳定摘要
- 用户主动要求“先想清楚”

**跳过**:
- 需求已完整且无歧义
- 单文件小修改
- 用户明确说“直接做”“just do it”“别问了”

## 提问格式

每次只问一个问题。

默认格式：

```text
<一句话问题>

1. <选项> — <含义>
2. <选项> — <含义>
3. <选项> — <含义>
N. 其他（请描述）

回复数字选择，可多选用逗号分隔（如 1,3）
```

规则：
- 优先使用收敛式选项，不给开放题
- 只有 `capability-catalog.yaml` 中没有足够选项时，才允许短文本补充
- 每轮只推进一个字段，不把多个问题塞在同一轮文本里

## 四阶段流程

### Phase 0: 能力匹配

目标：锁定最相关的 skill / capability。

流程：
1. 读取 `references/capability-catalog.yaml`
2. 用条目的 `triggers` 和 `user_examples` 匹配用户输入
3. 精确匹配到唯一条目 -> 直接锁定该 skill
4. 匹配到多个条目 -> 问用户选择
5. 没有明确匹配 -> 用最接近的 2-3 个 capability 做一次收敛式路由提问

硬规则：
- 只从 `capability-catalog.yaml` 读取匹配规则，不在 skill 里硬编码某个下游能力
- 如果条目存在 `template_index_path`，下一阶段进入模板匹配

### Phase 1: 模板匹配

仅对声明了 `template_index_path` 的 skill 生效。

流程：
1. 读取匹配 skill 的 `template_index_path`
2. 如果索引不存在、为空、或没有条目 -> 跳过本阶段，进入 Phase 2
3. 如果索引有条目，先问用户：
   - 直接用现有模板
   - 修改已有工作流
   - 新建工作流
4. 如果用户选 `existing` 或 `modify` 且还没指定模板，再让用户从索引条目里选模板
5. 如果最终选择是 `existing` 或 `modify`，直接输出摘要并结束
6. 如果最终选择是 `create`，继续进入 Phase 2

规则：
- `工作流ID` 与 `工作流名称` 必须从索引条目的 `id` / `name` 填入摘要
- `modify` 只是摘要模式；真正的 `update / fork` 确认留给下游 skill
- 这个模板匹配机制是通用的，不只服务 `cartooner-video`

### Phase 2: 需求澄清

目标：根据匹配条目的 `required_fields` 动态生成问题。

规则：
- 每个问题都来自 `required_fields[*].question`
- 每个问题的选项都来自 `required_fields[*].options`
- 用户初始消息里已经回答的字段直接跳过
- 用户直接丢素材、链接、截图、视频、文案时，视为对相关字段的回答或约束
- 用户催促“直接做”时，立即用当前已知信息进入 Phase 3
- 总问题数上限 7

### Phase 3: 输出需求摘要

规则：
- 如果 capability 条目定义了 `summary_contract`，按该合同输出
- 如果没有 `summary_contract`，输出通用摘要：

```markdown
## 需求摘要
**推荐 Skill**: <skill_id>
**已确认字段**:
- <字段1>: <用户选择或约束>
- <字段2>: <用户选择或约束>
**用户输入**: <原始自然语言>
```

- 输出摘要后，明确这份摘要可直接作为下游执行规格

## 通道适配

### 飞书通道

当可用飞书提问工具时：
- Phase 1 模板匹配使用卡片
- Phase 2 从 `required_fields` 动态生成卡片问题
- 每次只推进一个字段，除非同一阶段的多个字段明显可以合并且不会增加歧义

模板匹配卡片示例：

```json
{
  "questions": [
    {
      "question": "你想怎么处理这次视频需求？",
      "header": "选择工作流",
      "options": [
        { "label": "直接用现有模板", "description": "从已有工作流里选一个直接执行" },
        { "label": "修改已有工作流", "description": "先选模板，再决定 update 或 fork" },
        { "label": "新建工作流", "description": "从头设计一套新的" }
      ],
      "multiSelect": false
    }
  ]
}
```

### 非飞书通道

继续使用文本编号格式。

模板匹配问题示例：

```text
你想怎么处理这次需求？

1. 直接用现有模板 — 从已有工作流里选一个直接执行
2. 修改已有工作流 — 先选模板，再由下游决定 update 或 fork
3. 新建工作流 — 从头设计一套新的
N. 其他（请描述）
```

## 行为规则

- **catalog 驱动**：匹配、模板索引路径、需求字段、摘要合同都从 `capability-catalog.yaml` 读取
- **模板优先**：对声明了模板索引的 skill，先做模板匹配，再决定是否继续澄清
- **术语翻译**：用户只看到业务语义，不看到底层技术词
- **收敛式选项**：优先 2-3 个有意义的选项，不给泛泛开放题
- **行为即回答**：用户直接丢素材或示例时，优先把它解释成回答和约束
- **用户管是什么，Agent 管怎么做**：用户选平台、内容类型、风格；Agent 负责把它翻译成后续技术参数
- **容忍覆盖**：用户随时改主意，后续摘要以最新选择为准
- **不收素材**：素材收集属于下游执行 skill
- **先问后查**：摘要完成前，禁止读代码、提方案、改文件
- **与 ACP 委托衔接**：摘要可直接用于 ACP / 委托模板的 `Objective` / `Acceptance`
